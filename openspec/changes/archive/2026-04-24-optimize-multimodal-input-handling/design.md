## Context

当前 chat-service 的 /api_planning 接口已实现多模态输入处理，但存在以下问题：
- 视频检测走完整对话流程，生成不必要的 session
- 定时任务只返回单条语音，前端定时触发时无语音提示
- 语音和文字的处理规则需要明确统一

约束：
- 复用现有 TTS 服务（阿里云 DashScope）
- 保持向后兼容（前端可逐步适配）
- 视频检测响应时间 < 5s

## Goals / Non-Goals

**Goals:**
- 视频检测使用模板回复 + TTS，不存储 session
- 文字/语音输入检测定时任务后返回双语音
- 提醒语音使用自然表达映射

**Non-Goals:**
- 不支持图像输入
- 不改变现有的 session 管理逻辑
- 不改变前端定时任务触发机制

## Decisions

### 1. 视频检测回复使用模板而非 LLM

**决策**: 使用预定义模板生成回复，而非调用 LLM

**理由**:
- 视频检测是独立事件，不需要上下文理解
- 模板响应快速（< 100ms），LLM 需要 2-5s
- 回复内容可控，避免 LLM 生成不当内容

**模板映射**:
```python
VIDEO_RESPONSE_TEMPLATES = {
    "safe": "视频已记录，一切正常",
    "warning": "检测到潜在风险，请注意安全",
    "critical": "检测到摔倒，已通知紧急联系人"
}
```

### 2. 双语音返回机制

**决策**: 后端生成两条语音，一起返回前端

**理由**:
- 前端简单，只需播放和存储
- 语音质量一致（同一 TTS 模型）
- 离线时定时提醒仍可播放（语音已存储）

**实现**:
```python
{
    "audio_confirm": "好的，已设置每天早上8点提醒你吃药",  # 立即播放
    "audio_reminder": "该吃药了！",                       # 定时播放
    "schedule": {
        "cron": "0 8 * * *",
        "message": "提醒吃药",
        "audio_reminder": "base64..."  # 存储供前端使用
    }
}
```

### 3. 自然表达映射

**决策**: 使用预定义的自然表达映射，而非直接使用 message 字段

**映射表**:
```python
REMINDER_TEXT_MAP = {
    "medication_reminder": "该吃药了！",
    "meal_reminder": "吃饭时间到了，记得按时用餐",
    "care_check": "该做健康检查了",
    "custom": None  # 使用 message 字段
}
```

## Risks / Trade-offs

### 1. 响应体增大
**风险**: 双语音导致响应体增大（每条语音约 50-100KB）
**缓解**: 
- 语音使用压缩格式（OPUS）
- 前端及时释放不需要的音频对象

### 2. TTS 调用延迟
**风险**: 生成两条语音增加 1-2s 延迟
**缓解**:
- 异步生成语音（与 LLM 对话并行）
- 使用快速 TTS 模型

### 3. 模板回复不够灵活
**风险**: 固定模板无法适应所有场景
**缓解**:
- 提供 3 种风险等级的模板
- 未来可根据需要扩展模板
