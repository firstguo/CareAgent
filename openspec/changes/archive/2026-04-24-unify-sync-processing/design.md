## Context

当前架构存在两套处理流程：
1. **同步对话流程**：文本/语音 → Session 管理 → LLM → TTS → 返回
2. **异步任务流程**：视频 → 创建 task → 后台执行 → 前端轮询 → 返回

视频检测采用异步流程的原因：
- 视频处理耗时较长（5-12 秒）
- 避免 HTTP 请求超时

但实际使用中发现：
- 大部分视频是日常监控（90%+ 为 normal）
- 非危险视频不需要 LLM 处理和 session 存储
- 异步流程增加前端复杂度和后端维护成本

## Goals / Non-Goals

**Goals**:
- 统一所有输入类型为同步处理
- 视频检测前置，根据风险等级智能分流
- 非危险视频快速返回（不调 LLM）
- 危险视频文本化后走对话流程
- 移除异步任务流程

**Non-Goals**:
- 不改变 LLM 调用逻辑
- 不改变 Session 管理机制
- 不改变 TTS 生成逻辑
- 不支持流式响应（保持简单同步）

## Decisions

### Decision 1: 同步处理替代异步

**选择**：所有输入类型改为同步返回

**理由**：
- 视频最长 11 秒，在 HTTP 超时范围内（通常 30 秒）
- 前端简化，无需轮询逻辑
- 后端统一，降低维护成本

**风险**：
- 如果视频检测超时（>30 秒），请求失败
- 前端需要良好的加载提示（"正在分析视频..."）

**缓解措施**：
- 设置合理的超时时间（60 秒）
- 前端添加进度提示
- 监控视频检测耗时，优化性能

---

### Decision 2: 视频风险分流策略

**选择**：
```
视频检测
   │
   ├─ normal → 返回 ignored（不调 LLM）
   └─ critical/warning → 生成文本 → 走对话流程
```

**理由**：
- 90%+ 视频为 normal，节省 LLM 调用成本
- 危险事件需要人性化建议，LLM 更合适
- 危险事件融入对话上下文，支持后续跟进

**文本生成方式**：使用检测结果拼接（方案 A）

```python
def format_detection_as_text(detection_result):
    risk_level = detection_result["risk_level"]
    confidence = detection_result["confidence"]
    evidence = detection_result.get("temporal_evidence", "")
    
    risk_desc = {
        "critical": "严重风险（可能摔倒）",
        "warning": "潜在风险"
    }
    
    return f"【视频检测】{risk_desc.get(risk_level, risk_level)}，置信度{confidence:.0%}。时序证据：{evidence}"
```

**理由**：
- 无需额外 LLM 调用，节省成本
- 格式统一，易于调试
- 包含完整检测信息

---

### Decision 3: Session 处理

**选择**：危险视频创建新 session

**理由**：
- 危险事件独立会话，不混淆日常对话
- 便于后续跟进（如定时确认任务）
- 符合事件驱动的设计原则

**示例**：
```
用户："今天天气怎么样？"
  → session_1: "天气不错..."

视频摔倒检测（危险）
  → session_2: "检测到摔倒！建议立即..."
  → 独立会话
```

---

### Decision 4: 响应格式统一

**选择**：所有输入类型返回统一格式

```python
# 文本/语音/视频（危险）
{
  "mode": "conversation",
  "session_id": "sess_xxx",
  "response": "...",
  "audio_confirm": "base64...",
  "audio_reminder": "base64...",
  "schedule": {...}
}

# 视频（非危险）
{
  "mode": "ignored",
  "status": "ignored",
  "reason": "no_risk_detected",
  "result": {
    "risk_level": "normal",
    "confidence": 0.9
  }
}
```

**理由**：
- 前端只需处理两种模式：`conversation` 和 `ignored`
- 保持向后兼容（文本/语音响应格式不变）
- 清晰区分危险/非危险事件

---

### Decision 5: 移除异步任务流程

**选择**：删除 `handle_async_task` 和相关代码

**理由**：
- 所有输入改为同步，不再需要异步流程
- 简化代码库，降低维护成本
- 移除 MongoDB tasks 集合的写入（节省存储）

**影响**：
- `api_task_status` 端点不再使用（保留但标记废弃）
- `execute_task` 函数不再调用
- 前端无需轮询逻辑

---

## Risks / Trade-offs

### Risk 1: 视频检测超时

**风险**：视频检测耗时过长（>30 秒）导致 HTTP 超时

**影响**：请求失败，用户体验差

**缓解**：
- 监控视频检测耗时
- 优化抽帧策略（减少帧数或并行度）
- 设置合理的超时时间（60 秒）
- 前端添加超时重试机制

---

### Risk 2: 前端等待体验

**风险**：用户等待 5-11 秒，体验不佳

**影响**：用户可能认为系统卡死

**缓解**：
- 前端显示进度提示（"正在分析视频..."）
- 添加加载动画
- 可选：显示检测进度（"已处理 3/5 帧"）

---

### Risk 3: 误判风险

**风险**：视频检测误判（normal 但实际危险）

**影响**：危险事件被忽略

**缓解**：
- 调低 critical/warning 阈值（保守策略）
- 提供手动触发 LLM 分析的入口
- 定期人工复核检测结果

---

### Trade-off: 同步 vs 异步

**选择同步的代价**：
- 前端需要等待较长时间
- HTTP 连接占用时间增加

**选择同步的收益**：
- 前端代码简化 50%+
- 后端代码简化 30%+
- 非危险视频成本降低 70%+
- 危险事件处理更人性化

**结论**：收益 > 代价，选择同步
